# Phase 1 PostgreSQL Schema for PathGuard

## SQL Code (For Supabase)

```sql
-- 1. USERS TABLE
CREATE TABLE users (
    uid UUID REFERENCES auth.users NOT NULL PRIMARY KEY,
    email TEXT UNIQUE NOT NULL,
    role TEXT DEFAULT 'user' CHECK (role IN ('user', 'admin')),
    created_at TIMESTAMPTZ DEFAULT NOW() NOT NULL
);

-- Enable RLS
ALTER TABLE users ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Users can read own data" ON users FOR SELECT USING (auth.uid() = uid);

-- 2. TRANSACTIONS TABLE
CREATE TABLE transactions (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    sender_id UUID REFERENCES users(uid) NOT NULL,
    recipient_id UUID REFERENCES users(uid) NOT NULL,
    amount_cents INTEGER NOT NULL CHECK (amount_cents > 0),
    currency TEXT DEFAULT 'USD' NOT NULL,
    status TEXT DEFAULT 'pending' CHECK (status IN ('pending', 'processing', 'completed', 'failed', 'flagged')),
    path_hash TEXT NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW() NOT NULL
);

-- Enable RLS
ALTER TABLE transactions ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Users view own transactions" ON transactions FOR SELECT USING (auth.uid() = sender_id OR auth.uid() = recipient_id);
CREATE POLICY "Users insert own transactions" ON transactions FOR INSERT WITH CHECK (auth.uid() = sender_id);
CREATE POLICY "Admins update status" ON transactions FOR UPDATE USING (
    EXISTS (SELECT 1 FROM users WHERE uid = auth.uid() AND role = 'admin')
);

-- 3. AUDIT LOGS TABLE (IMMUTABLE LEDGER)
CREATE TABLE audit_logs (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    transaction_id UUID REFERENCES transactions(id) NOT NULL,
    action TEXT NOT NULL,
    actor_id UUID REFERENCES users(uid) NOT NULL,
    timestamp TIMESTAMPTZ DEFAULT NOW() NOT NULL,
    previous_hash TEXT NOT NULL,
    current_hash TEXT NOT NULL
);

-- Enable RLS (Read-only for everyone with access, insert-only via secure functions)
ALTER TABLE audit_logs ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Anyone can read audit logs" ON audit_logs FOR SELECT USING (true);
CREATE POLICY "No one can update audit logs" ON audit_logs FOR UPDATE USING (false);
CREATE POLICY "No one can delete audit logs" ON audit_logs FOR DELETE USING (false);

-- Trigger to absolutely block UPDATE and DELETE on audit_logs at the database level
CREATE OR REPLACE FUNCTION prevent_audit_log_modification()
RETURNS TRIGGER AS $$
BEGIN
    RAISE EXCEPTION 'Audit logs are immutable. UPDATE and DELETE operations are strictly forbidden.';
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER enforce_audit_log_immutability
BEFORE UPDATE OR DELETE ON audit_logs
FOR EACH ROW EXECUTE FUNCTION prevent_audit_log_modification();

-- 4. INCIDENTS TABLE
CREATE TABLE incidents (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    transaction_id UUID REFERENCES transactions(id) NOT NULL,
    risk_type TEXT NOT NULL,
    severity TEXT CHECK (severity IN ('low', 'medium', 'high', 'critical')),
    resolved_by UUID REFERENCES users(uid),
    resolved_at TIMESTAMPTZ,
    created_at TIMESTAMPTZ DEFAULT NOW() NOT NULL
);

-- Enable RLS
ALTER TABLE incidents ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Users can see incidents on own transactions" ON incidents FOR SELECT USING (
    EXISTS (SELECT 1 FROM transactions WHERE id = incidents.transaction_id AND (sender_id = auth.uid() OR recipient_id = auth.uid()))
);
CREATE POLICY "Admins manage incidents" ON incidents FOR ALL USING (
    EXISTS (SELECT 1 FROM users WHERE uid = auth.uid() AND role = 'admin')
);
```

## Plain English Explanation
*   `users`: Stores profile data. Connects to Supabase Auth (`auth.users`) for secure login.
*   `transactions`: Stores the core money movement. `amount_cents` uses integers to prevent math errors common with decimals. Policies ensure users only see their own money.
*   `audit_logs`: The immutable heart of the system. We use a cryptographically enforced trigger `prevent_audit_log_modification()` that makes it physically impossible for anyone (even a breached frontend) to alter or delete a log entry once written. `previous_hash` and `current_hash` create a verifiable chain: if someone alters log #2, the hash for log #3 suddenly becomes mathematically invalid, instantly proving tampering occurred. 
*   `incidents`: Stores risk events generated by the AI Engine.

## ⚠️ Warning: Backend Cloud Functions Required
Writing to `audit_logs` **should NOT be done via direct client writes (from the browser/frontend).**
Because `current_hash` relies on server-side secrets or exact timestamps to create a valid cryptographic hash linking to `previous_hash`, this logic *must* happen in a secure backend environment (like Supabase Edge Functions, standard PostgreSQL functions, or a dedicated Node.js backend). If the frontend generates the hash, a compromised client could forge a fake transparent trail.
