# PathGuard Backend API: Route Specifications

## Overview
This document defines the core Go/Java API endpoints required for the PathGuard Phase 1 State Pilot. It is designed to be highly modular so you can prompt an AI to build these routes one-by-one.

All endpoints must be placed behind an **API Gateway** enforcing TLS 1.3 and rate limiting.

---

## 1. Authentication Middleware (The ID)
Before any route is hit, the request must pass through an Authentication Middleware.
- **Header:** `Authorization: Bearer <JWT_ACCESS_TOKEN>`
- **Logic:** The middleware must verify the JWT signature using the Identity Provider's JWKS (Public Key). If invalid, expired, or missing, it must return `401 Unauthorized`.
- **RBAC Check:** If the user is accessing `admin` routes, verify the JWT claims contain `"role": "admin"`.

---

## 2. The Core API Routes (Phase 1)

### Route A: Initiate Transfer
**Endpoint:** `POST /api/v1/transfers`
**Purpose:** Create a new pending transaction between two users.

**Security Constraints:**
1. Verifies the sender's account balance.
2. Validates the `recipient_id` exists and is KYC-Verified.
3. Cryptographically verifies the `client_signature` (The ECDSA signature generated by the user's mobile Secure Enclave).
4. **Action:** Inserts standard status `INITIATED` into the `transactions` table.

**Prompt for Developer/AI:**
```text
Task: Build the POST /api/v1/transfers endpoint in Go (Gin) or Java (Spring Boot).
Requirements:
1. Accept a JSON payload with `amount`, `recipient_id`, and `client_signature`.
2. Extract the `sender_id` securely from the JWT token in the Auth header (do NOT trust a sender_id in the JSON body).
3. Validate the `client_signature` against the sender's stored public key.
4. Begin a Database Transaction. Insert into the `transactions` table and the `audit_ledger` table. Commit or Rollback. Provide clean error handling.
```

### Route B: Get Transfer Path
**Endpoint:** `GET /api/v1/transfers/{tx_id}/path`
**Purpose:** Returning the visual hops to the user frontend for the "Live Tracker" UI.

**Security Constraints:**
1. **Authorization:** The API must verify that the `usr_id` inside the JWT token matches EITHER the `sender_id` OR the `recipient_id` of the transaction. If it's a random user querying the ID, return `403 Forbidden`.

**Prompt for Developer/AI:**
```text
Task: Build the GET /api/v1/transfers/{tx_id}/path endpoint in Go or Java.
Requirements:
1. Use an ORM or safe parameterized SQL queries to prevent SQL Injection.
2. Write a strict Authorization layer verifying the requesting User ID (from the JWT) owns or is receiving this transaction.
3. Fetch the transaction summary and a joined list of rows from the `transaction_hops` table, ordered by `hop_sequence`.
4. Return a clean JSON response that the Flutter frontend can easily map to a timeline UI.
```

### Route C: Webhook Receiver (Bank Status Update)
**Endpoint:** `POST /api/v1/webhooks/bank/status`
**Purpose:** External banks will call this URL to tell PathGuard a specific hop has "CLEARED" or "FAILED".

**Security Constraints:**
1. **HMAC Signature Check:** The pre-handler middleware MUST calculate the HMAC-SHA256 signature of the raw request body using the shared Webhook Secret (stored in AWS KMS). This signature must mathematically match the `x-bank-signature` HTTP header.
2. **Idempotency:** The payload must include an `event_id` or `timestamp`. Block replays older than 5 minutes.

**Prompt for Developer/AI:**
```text
Task: Build a highly secure Webhook Receiver endpoint: POST /api/v1/webhooks/bank/status.
Requirements:
1. Write a custom middleware that reads the raw HTTP body bytes, generates an HMAC-SHA256 hash using a secret key (loaded from ENV/KMS), and compares it using a timing-safe string comparison to the `x-bank-signature` header.
2. If the signature is valid, parse the JSON. It will contain `hop_id` and `new_status`.
3. Read from a Redis cache to prevent processing the identical webhook twice (Idempotency).
4. Update the `transaction_hops` table and append a record to the `audit_ledger` table.
```

## Conclusion
By breaking the API down into `Initiate`, `Query`, and `Webhook`, the system remains simple enough to prompt local AI models to build one clean, highly secure file at a time without overwhelming their context window.
